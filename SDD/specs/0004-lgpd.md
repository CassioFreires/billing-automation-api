# Spec 0004 — Módulo de LGPD (direitos do titular)

- **Autor**: —
- **Data**: 2026-07-01
- **Roadmap**: PR-06 (parte de código)

> ⚠️ **Não é aconselhamento jurídico.** Esta spec cobre a **parte técnica** (direitos do titular via API). As obrigações legais/documentais (base legal, política de privacidade, termos, DPA, encarregado/DPO) precisam de revisão jurídica — ver §11.

## 1. Problema / Motivação

O sistema processa **dados pessoais** de clientes (nome, telefone, CPF/CNPJ, dívida). A LGPD garante ao titular direitos de **acesso/portabilidade** (Art. 18, II/V) e **eliminação** (Art. 18, VI). Precisamos oferecer esses direitos por API, respeitando a **retenção legal** de registros financeiros (Art. 16, I).

## 2. Objetivo

Endpoints (por tenant, autenticados) para:
- **Exportar** todos os dados de um titular (cliente + faturas) — acesso/portabilidade.
- **Anonimizar** um titular — remove os dados pessoais (nome, telefone, documento) mas **preserva as faturas** (retenção fiscal/contábil), tornando o cliente não reidentificável.

**Fora de escopo**: consentimento granular/opt-in, verificação de identidade do titular, exclusão física total (o `DELETE /clients/:id` já existe e apaga em cascata — mas anonimização é o caminho recomendado para billing), trilha de auditoria completa.

## 3. Regras de negócio

- **RN-L1**: Exportar retorna o cliente + suas faturas, **escopado por tenant** (nunca cross-tenant).
- **RN-L2**: Anonimizar sobrescreve `name`/`phone`/`document` por valores não identificáveis e marca `anonymizedAt`. As `Invoice` permanecem (retenção legal), agora ligadas a um titular anônimo.
- **RN-L3**: O `phone` anonimizado precisa continuar **único por tenant** — usa-se um placeholder único (`anon-<id>`).
- **RN-L4**: Anonimizar é **idempotente** — cliente já anonimizado não é reprocessado.
- **RN-L5**: Só usuários autenticados do tenant podem exercer esses endpoints sobre os próprios clientes.

## 4. Impacto no modelo de dados

- `Client.anonymizedAt DateTime?` — marca quando o titular foi anonimizado. Migração aditiva/idempotente.

## 5. Contrato de API

```
GET  /api/lgpd/clients/:clientId/export      (JWT)
  → 200 { exportedAt, client, invoices }     | 404

POST /api/lgpd/clients/:clientId/anonymize   (JWT)
  → 200 { anonymized: true, client }         | 404
```

## 6. Fluxo

**Export**: `findById` (escopado) → se não achar 404 → busca faturas do cliente (escopadas) → retorna estrutura JSON.
**Anonymize**: `findById` (escopado) → se já `anonymizedAt`, retorna (idempotente) → senão sobrescreve PII + `anonymizedAt`.

## 7. Camadas afetadas

- [ ] Schema/migration — `Client.anonymizedAt`
- [ ] Repository — `ClientRepository.anonymize`; `InvoiceRepository.findByClientId` (escopado)
- [ ] Service — `LgpdService` (export/anonymize), com injeção de dependência
- [ ] Controller/Router — `/api/lgpd` (JWT)
- [ ] Testes — export (ok/404), anonymize (ok/idempotente/404)

## 8. Critérios de aceite

- [ ] Export retorna cliente + faturas do tenant; 404 se não existir/for de outro tenant.
- [ ] Anonymize remove PII (nome/telefone/documento) e mantém as faturas.
- [ ] Anonymize é idempotente.
- [ ] Telefone anonimizado não colide com a unique por tenant.

## 9. Riscos / considerações

- **Retenção x eliminação**: anonimizamos em vez de apagar para manter registros financeiros (Art. 16). Se o titular exigir eliminação total e não houver obrigação de retenção, usar o `DELETE /clients/:id`.
- **Reidentificação**: garantir que nenhum outro campo permita reidentificar (hoje o cliente não tem e-mail; faturas guardam valores/datas, não PII).
- **Auditoria**: registrar quem/quando pediu exportação/anonimização é um follow-up recomendável.

## 10. Notas de implementação

- Anonimização: `name = 'Titular anonimizado (LGPD)'`, `phone = 'anon-<id>'`, `document = 'ANONIMIZADO'`, `anonymizedAt = now()`.
- `LgpdService` reusa `ClientRepository`/`InvoiceRepository` (escopo por tenant já garantido pelo contexto).
